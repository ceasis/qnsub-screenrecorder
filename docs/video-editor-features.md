# Video Editor — Feature Catalog

## Recent projects rail

Appears on the left of the empty state and is populated by merging three sources: a persistent JSON index at `userData/recentProjects.json` that tracks every project you've ever recorded/saved/loaded, a folder scan of your configured save folder, and a folder scan of the system folders (Desktop, Downloads, Videos, Documents on Windows). Each card shows the source video filename, a relative timestamp ("3m ago / 2h ago / 3d ago / Apr 8"), and a parent-folder/project-folder path that truncates with ellipsis. Hovering a card reveals a small × button that removes it from the recents list without touching the project file on disk. The list re-renders its time strings every 30 seconds so "just now" stays accurate while the user lingers, and dead/moved files get auto-pruned from the persistent index on every read. Both `.json` (current) and `.xml` (legacy stub) project files show up; the loader transparently upgrades XML stubs by probing the sibling MP4 and rebuilding a fresh `EditorProject` from `emptyProject()`.

## Upload + auto-process

Kicks off the entire editing flow with a single Upload Video button that opens a native file picker, probes the chosen file for duration/dimensions/fps via ffmpeg's stderr, and creates an `EditorProject` with one full-length clip. Immediately after upload, the editor automatically runs ffmpeg's `silencedetect` filter against the configured threshold (default -30dB / 0.5s) and renders the detected silent regions as yellow striped overlays on the timeline. This gives the user instant value out of the upload before they touch any controls, in line with the spec's "user does NOT need a tutorial" goal. The detection runs again on demand via the Re-scan silence button or whenever the threshold/min-duration sliders change in the Silence settings popover. Detection uses regex parsing of ffmpeg's `silence_start:` / `silence_end:` stderr output rather than a separate ffprobe pass.

## Single timeline + draggable playhead

The unified editing surface — clips render as colored bands at their source-time positions, with deleted regions showing as gaps and silence regions as yellow stripes underneath. A single red playhead handle is always visible and can be grabbed directly or via clicking anywhere on the timeline; document-level mouse listeners keep scrubbing smooth even when the cursor leaves the timeline element. The timeline has a Zoom slider (0.5×–8×) and the timeline element is locked to 120px tall with `flex: 0 0 120px` so it doesn't collapse inside the flex column. Click a clip to select it, click empty space to deselect, and the per-clip playback rate live-updates the `<video>` element so what you see during scrubbing matches the export. During playback, the editor automatically skips removed (ghost) regions by jumping to the next non-removed clip's start.

## Editing actions

The standard cut-trim-delete primitives: Split (S key) cuts the playhead-containing clip into two pieces, Trim before drops everything before the playhead (clipping the containing clip if needed), Trim after does the symmetric thing, and Delete removes the selected clip entirely. Every editing action commits to a Reducer-based undo/redo history capped at 80 levels, with Ctrl+Z / Ctrl+Shift+Z (or Ctrl+Y) bound globally and tied via the `useEditorState` hook. The history distinguishes between `commit` (push to undo stack) and `set` (ephemeral patch, no history) so things like silence detection results don't pollute the undo stack. All edit transforms (`splitClipAt`, `trimBefore`, `trimAfter`, `deleteClip`, `applySilenceAsGhostGaps`) are pure functions exported from `useEditorState.ts` so they're testable and re-usable. The selected clip and selected text overlay each have their own keyboard delete handling.

## Auto silence trim with ghost gaps

The killer feature — once silence is detected, clicking ✂ Auto-trim silence splits the existing clips at every silence boundary and marks the inner pieces as `removed: true` instead of deleting them. Removed clips render with diagonal hatching plus a tiny ↺ restore button on the band itself, and a separate ↺ Restore all button in the toolbar un-removes everything in one click. The Silence settings popover lets you tweak the threshold (-60 to -10 dB) and min duration (0.2-2.0 s) and re-scan on the fly. Removed clips are excluded from export but kept in the project file so they're fully restorable across save/load cycles. The export pipeline filters them out by `c.removed === false` before building the ffmpeg filter graph.

## Per-clip inspector

Appears below the timeline whenever a clip is selected and exposes Label (text input shown inline on the band), Speed (chip selector with 0.5/1/1.5/2/4× presets plus ± buttons for arbitrary 0.25× steps clamped to 0.25-4×), Volume slider 0-200%, Mute toggle, Fade in (0-3s), and Fade out (0-3s). Speed adjustments use ffmpeg's `setpts=PTS/<speed>` for video and chained `atempo` filters for audio (atempo only supports 0.5-2 in a single step so chains handle extreme values). Fade-in/out are baked via the `fade` and `afade` filters at the clip's local time origin after the speed adjust. Mark removed / Restore + a destructive Delete button live in the inspector for the selected clip too. The Volume display rounds to 2 decimals and the output duration calculation accounts for per-clip speed.

## Text overlays + templates

Supports Title (centered big bold), Lower third (bottom-left with semi-opaque background bar), CTA (centered with red pill background), and Plain (small centered) presets that drop a text element at the playhead with a 4-second visibility window. Each overlay stores position and size as 0..1 fractions of the source frame, drag-to-position works directly on the preview with a corner resize handle, and double-click enters inline text editing mode. The text inspector exposes content (textarea), color (native picker), font size (slider as fraction of frame height), alignment (left/center/right), and start/end times with "Set to playhead" buttons. During preview, active overlays render at full opacity and inactive ones show at 35% with a dashed outline so you can still grab them. On export, each overlay is rasterized via offscreen canvas to a PNG at the source resolution and burned into the output via ffmpeg `overlay` filters with `enable='between(t,start,end)'` mapped from source-time to output-time.

## Audio panel + BGM

Supports adding one background music track via a native audio file picker, with volume control (0-100%) and an auto-duck toggle that lowers the BGM whenever the voice is present. Ducking uses ffmpeg's `sidechaincompress` filter with the voice as the sidechain input, mixed back with the unducked voice via `amix`. The BGM is looped via `aloop` so it covers any output length, and a Noise reduction toggle adds an `afftdn` filter to the voice chain when enabled. Per-clip mute and volume cover the "mute segments" requirement — splitting at the boundary then muting the inner clip is the workflow. The Audio panel uses the standardized two-column row layout with help icons.

## Export pipeline + presets

Opens a modal with platform presets (YouTube 16:9, YouTube Shorts 9:16, TikTok 9:16, Instagram Reel 9:16, Instagram 1:1, Custom) and a Resolution selector (720p / 1080p that scales the chosen preset). The export builds a single `filter_complex` graph that handles trim/setpts/fade per active clip → concat → resolution scale → text overlay PNG burns → optional BGM mix with optional sidechain ducking → optional `afftdn` denoise → AAC + libx264/NVENC/VideoToolbox encode. Hardware encoder selection follows the same platform-aware fallback as the recording finalize. A live progress bar parses ffmpeg's stderr `time=` lines, computes percent against the precomputed total output duration, and a Cancel button kills the active ffmpeg child process via `SIGKILL`. Temp PNG files for text overlays are written to `app.getPath('temp')` and cleaned up in `finally` regardless of success.

## Project save/load + auto-save

Persists every change via a 600ms-debounced auto-save once a project path is set, writing pretty-printed JSON to disk. The Save button explicitly writes a `project_<stamp>.json` next to the source video if no project path exists yet. Open Project loads any saved JSON file via a native picker and auto-rehydrates the entire editor state (clips, text overlays, BGM, silence detection results, export settings). The recording handler creates a stub project alongside every new recording so the file is ready to open the moment you switch to the editor tab. Both the editor save/load and the recording save touch the persistent recents index so the rail stays in sync without explicit user action.

## Playback controls

A dedicated transport row sits under the preview with Start (⏮), Back 1s (◀◀), Play/Pause (▶/❚❚), Forward 1s (▶▶), and End (⏭) buttons for step-accurate scrubbing. The Space bar toggles play/pause globally and the ArrowLeft/ArrowRight keys nudge the playhead by 0.1s (or 1s while holding Shift). A time readout shows the current source time, the source duration, and the computed output duration (which accounts for per-clip speed and removed ghost gaps). Seek clamps automatically to [0, duration] so you can't scrub past the ends, and all three of click-to-seek on the timeline, transport-button seeks, and keyboard nudges run through the same `seek()` function. The `<video>` element uses its native controls hidden so all playback flows through our React state.

## Fade transitions between clips

Each clip has independent Fade in (0-3s) and Fade out (0-3s) sliders in the inspector that bake into the export via ffmpeg's `fade` and `afade` filters applied after the trim/speed pass. Cross-clip fades fall naturally out of this — fading out the end of clip A plus fading in the start of clip B produces a proper crossfade since the concat filter stitches them back-to-back. Fade durations are clamped to the clip's actual output length so a 3s fade on a 1s clip just becomes a 1s fade. Both video and audio fades are applied symmetrically so voice/music fade together with the picture. The inspector live-updates as you drag the slider; the effect is baked, not previewed in the player.

## Auto cut suggestions with accept/reject

Silence detection is currently the only rule-based suggestion — detected silent regions render as yellow striped overlays on the timeline so you can see exactly what would be cut before committing. "Auto-trim silence" is the bulk-accept action and converts every silent region into a removed ghost gap in one click. The ↺ restore button on each removed clip is the per-region reject, and ↺ Restore All un-rejects everything. Silent regions that you've already trimmed stay visible as ghost gaps so you can always change your mind later. Additional suggestion types (low-energy regions, filler words, repeated phrases) are deliberately out of scope for the current rule-based release since they need speech recognition.

## Async processing

All ffmpeg work (silence detect, probe, export, recording finalize) runs in a `child_process.spawn` subprocess so the Electron renderer stays responsive the whole time. A `busy` state in the editor shows "Analyzing audio…" / "Exporting…" / "Loading project…" in the toolbar, and the Cancel button on the export modal kills the active export process via `SIGKILL` through the `editor:cancelExport` IPC. Only one ffmpeg export can run at a time — the second invocation will refuse until the first finishes or is cancelled (single-slot in-memory queue rather than a full job scheduler). Progress is streamed back via the `editor:exportProgress` IPC channel as soon as each `time=HH:MM:SS.xx` line lands on ffmpeg's stderr. There's no persistent job queue that survives app restarts; cancelled or crashed exports are forgotten on next launch.

## User settings

Configurable preferences live inside each project file: silence threshold (-60 to -10 dB) and minimum duration (0.2-2.0s) for the detector, background music volume and ducking toggle, noise reduction on/off, export preset (YouTube/TikTok/Instagram variants plus Custom), and output resolution (720p/1080p). Settings are persisted to the project JSON alongside the clips and text overlays so re-opening a project restores exactly the same preferences you had last time. Strong defaults mean most users never touch settings — the auto silence run on upload uses -30dB / 0.5s which works for typical screen recordings. No global "Preferences" dialog: settings live where they apply. The threshold popover is a native `<details>` element for low-chrome discoverability.

## Storage system

Every recording creates its own `ScreenRecording_YYYYMMDD_HHMMSS/` folder under the user's save root containing `original_*.mp4` and `project_*.json` side by side — one folder per recording keeps things self-contained and easy to delete. The project JSON holds all edit metadata (clips, text overlays, markers, BGM, silence, settings) and references the source MP4 by absolute path. A persistent index at `userData/recentProjects.json` remembers every project the app has ever touched so you can find it later even if it's moved; dead entries get auto-pruned on each read. Everything is plain local files — no cloud, no database, no login, nothing sent over the network. Legacy XML project stubs from older builds get upgraded transparently on load.

## Simple UI layout + minimal controls

The editor follows a vertical flow: top toolbar (Upload / Open Project / Text / Silence / Undo / Save / Export) → video preview (capped at 45vh so the timeline stays visible without scrolling) → edit toolbar (Split / Trim before / Trim after / Delete / Restore / Silence settings) → timeline (120px tall, always visible) → selected-item inspectors (clip / text / audio panels) stacked below. The top toolbar exposes only the essential actions the spec asks for — Upload Video, Auto Edit (via Auto-trim silence), Add Text, Export — with secondary actions tucked behind contextual panels. All controls use the same two-column row pattern with right-aligned labels and capped 50% slider/dropdown widths for visual consistency with the Screen Recorder tab. The empty-state screen is a clean left-rail + centered upload card with zero other chrome. No tabs, modes, or hidden menus — everything the user can do is on-screen at once.

## Rule-based editing

Every automated decision the editor makes is driven by deterministic ffmpeg filters and pure functions — there are no AI models, no network calls, no stochastic behavior. Silence detection uses ffmpeg's `silencedetect` with user-configurable amplitude threshold and minimum duration. The silence-to-ghost-gap transform in `applySilenceAsGhostGaps` is a pure function that splits clips at every region boundary, so running it twice is idempotent. The undo/redo history stores full project snapshots rather than deltas so replaying state is always exact. This keeps the app reliable, offline-friendly, testable, and free of per-API-call costs — matching the spec's "prioritize deterministic logic instead of AI models" principle.

## Preview mode

**Not yet implemented.** The editor currently plays the full-resolution source MP4 directly in the `<video>` element, which works fine for typical screen recordings (they're already at 1920x1080 30fps) but may stutter on very large or high-bitrate files. A future addition would generate a lower-resolution proxy via a one-time ffmpeg scale pass at upload time and swap it in transparently for the preview player while keeping the original MP4 as the export source. Included here for completeness since it's on the requirements list.
